# Example 1: Base Simulation (Case 0)

This scenario demonstrates the simplest possible CosimGym layout: a one-federation instance connecting a math model (`spring_mass_damper`) to a driving force (`inputs4spring`).

## Scenario Source
Found at `src/scenarios/simple_test.yaml`.

## Core Takeaways

1. **Topology Definition:**
We declare a single `federation1` running a `zmq` broker.

2. **Model Instantiation:**
Inside the federate `Fed_simple`, two models are spawned:
```yaml
model_configs:
  - model_name: "spring_mass_damper"
    instantiation:
      n_instances: 1
    parameters:
      mass: 10.0
      spring_constant: 50.0
      damping_coefficient: 5.0

  - model_name: "inputs4spring"
    instantiation:
      n_instances: 1
    parameters:
      target_amplitude: 1.0 # The displacement the spring should reach
```

3. **Publish / Subscribe Mapping:**
The spring must read the force generated by the inputs, and the inputs might read the current state of the spring.
Notice how the HELICS strings align: 
- `inputs4spring` publishes `federation1/input4spring1/output_var`.
- `spring_mass_damper` subscribes `input_var` to `federation1/input4spring1/output_var`.

## Execution
Run it:
```bash
python src/test_script.py --scenario simple_test
```

Once complete, open the dashboard. You will see an under-damped harmonic oscillation graph where `spring_mass_damper_1_displacement` rings and eventually dampens to the `target_amplitude` generated by `inputs4spring_1_output`.
